## Roadmap

### 🟡 Next — Phase 2 Completion (SmartRouter)

**Goal**: Full per-project routing control and budget enforcement
**Unlocked by**: Current `request_logs` table + routing engine

#### Per-Project Routing Rules UI
- Quality tier selection per project (economy / balanced / performance)
- Task-level overrides: "always use performance for coding tasks"
- Provider allowlist/blocklist per project
- Stored in DB (`projects.routingConfig` jsonb column)
- Routing engine reads rules at request time (KV-cached)

#### Budget Enforcement
- Daily + monthly spend ceiling per project
- When ceiling hit: block request (return 429) OR downgrade to cheapest available
- Real-time budget check in routing engine from `request_logs` aggregates
- Budget alert when 80% / 100% of limit consumed

#### Routing Decisions Table
- Dashboard "Routing Decisions" tab: per-request log with task type, model selected, savings, latency
- Filterable by date, project, model, task type
- CSV export
- 90-day retention with configurable cron pruning

#### `POST /v1/embeddings`
- Passthrough to OpenAI text-embedding-3-small/large or Google embedding models
- Route by cost (embeddings have no quality tier — cheapest capable wins)
- Log to `request_logs` with `task_type = 'embedding'`

#### Request Log Retention Cron
- Nightly cron: `DELETE FROM request_logs WHERE created_at < NOW() - INTERVAL '90 days'`
- Configurable per org (enterprise: longer retention)

### 🔵 Phase 3 — Repo-Linked Attribution

**Goal**: Connect spend → routing decisions → actual code files
**Requires**: Phase 2 data in `request_logs`

#### GitHub Repo Integration
- Link GitHub repos to projects (Settings UI already prepped)
- OAuth GitHub app or PAT per org
- Code scanner: GitHub Search API finds model call sites (`openai.chat.completions.create`)
- Map `x-sr-callsite` header (optional `X-Source-File: file:line` from app) to repo location

#### Code Attribution Dashboard
- "Code Attribution" view: spend drilled down to repo → file → function
- Top 10 files by cost, with current model and suggested alternative
- Historical trend per file

#### Automated Recommendations
- Haiku analyzes top spending files + routing data → file-specific recommendation
- "job_analysis.py:406 — switch `gpt-4o` → `claude-haiku-4-5`, save $44/mo"
- One-click GitHub PR description draft (Haiku writes the PR body)

---

### 🔵 Phase 4 — Live Pricing + Quality Benchmarks

**Goal**: Pricing always current; routing decisions improve from real performance data
**Can be built in parallel with Phase 3**

#### Live Pricing Updater
- Cron (every 6h): fetch from LiteLLM pricing OSS JSON + provider APIs
- Store in `model_pricing` table (add to schema)
- Routing engine reads from DB instead of hardcoded constants
- "Model News" panel: new model auto-evaluated and added to catalog
- Pricing change alert if any model changes > 20%

#### Benchmark Continuous Learning
- Periodic evaluation on standard tasks (MMLU subset, HumanEval, GSM8K)
- Quality scores updated weekly (rolling average)
- Per-task benchmark: don't use overall MMLU score for coding-specific routing
- Quality score editor in dashboard (admin can override benchmark scores)

---

### 🔵 Phase 5 — Advanced Routing

**Goal**: Production-grade routing with streaming, fallbacks, and A/B testing

#### Fallback Chains
- If primary provider returns 429/5xx, auto-retry on secondary
- Per-project fallback order (e.g. OpenAI → Anthropic → Google)
- Retry budget: max 2 retries, within 10s total timeout
- Log fallback events separately for debugging

#### Load Balancing
- Round-robin across multiple provider keys for same provider
- Prevents single-key rate limiting for high-volume orgs
- Key selection weighted by remaining rate-limit budget

#### Latency-Aware Routing
- Track p50/p95 latency per (provider, model) from `request_logs`
- Add latency factor to routing score (configurable weight vs. cost)
- "Latency-optimized" quality tier in addition to cost-optimized

#### A/B Routing
- Admin defines experiment: 50% to model A, 50% to model B
- Dashboard: side-by-side cost, latency, (manual) quality rating
- Automatic winner selection after N requests

#### Prompt Caching
- Detect repeated system prompts (hash prefix)
- Enable provider-side caching automatically (Anthropic cache_control, OpenAI prompt caching)
- Report cache hit rate + actual savings from caching in dashboard
- Store cache_read_tokens separately (already in schema: `cacheReadTokens`)

---

### 🔵 Phase 6 — OpenRouter / LiteLLM Compatibility

**Goal**: Drop-in replacement for existing OpenRouter and LiteLLM users

#### OpenRouter Compatibility
- Accept `HTTP-Referer` and `X-Title` headers (OpenRouter convention)
- Support `provider/model` syntax: `openai/gpt-4.1-nano`, `anthropic/claude-haiku-4-5`
- Import from OpenRouter: paste OR key, pull routing history, suggest rules
- Side-by-side comparison view: SmartRouter vs OpenRouter cost for same traffic

#### LiteLLM Compatibility
- LiteLLM virtual model names as aliases in our catalog
- `litellm/` prefix support
- Config file import: paste `litellm_params`, we ingest the routing config

#### Provider Expansion
- **Groq** — ultra-low latency, Llama 3.x / Mixtral (translator already in progress)
- **Mistral** — European data residency option, strong coding models
- **Azure OpenAI** — enterprise accounts with private deployments
- **AWS Bedrock** — Claude + Llama via AWS, enterprise customers
- **Cohere** — embedding specialist (Command R for RAG)
- **Local / private endpoints** — self-hosted Ollama, vLLM via custom base_url

---

### 🔵 Phase 7 — Enterprise

**Goal**: Compliance, multi-team, SLA, on-prem option

#### Compliance & Security
- SOC 2 Type II controls (access logs, encryption at rest, key rotation)
- GDPR: data residency options (EU Neon region, EU Vercel region)
- PII scrubbing in request logs (configurable regex patterns)
- Key rotation: re-encrypt DEK on schedule, invalidate old keys

#### Identity & Access
- SSO via Clerk (SAML, OIDC)
- SCIM provisioning (auto-sync from Okta / Azure AD)
- IP allowlist per org
- API key scoping: read-only tokens for dashboard integrations

#### SLA & Operations
- 99.9% uptime target
- < 50ms routing overhead p99
- Status page (public)
- On-prem / VPC deployment option (Docker Compose + self-managed Neon)

#### Billing & Spend Controls
- Volume discounts (negotiated per enterprise)
- Invoice billing (Net 30)
- Org-level spend caps with real-time enforcement
- Spend alerts to Slack / PagerDuty / webhooks (not just email)

## Near-Term Infrastructure

These are cross-cutting improvements that should ship before or alongside Phase 3:

### Webhook Delivery
- Org-configurable webhooks: `POST {url}` on alert, budget breach, routing decision
- Payload: same structure as email alerts
- Retry with exponential backoff (3 attempts)
- Webhook log in audit trail

### Slack Integration
- `/connect-slack` OAuth flow
- Alert notifications to selected channel
- Daily brief as Slack message (alternative to email)
- Slash command: `/aicostcentral spend` → inline summary

### Mobile Improvements
- Deep link from push notification → specific alert in dashboard
- Cost trend widget (home screen)
- Biometric auth for mobile settings

### Vercel KV → Upstash Redis Migration
- Current: Vercel KV (based on Upstash, but Vercel-managed, fewer features)
- Target: Direct Upstash Redis (rate limiting, pub/sub for real-time dashboard, better TTL control)
- Timeline: before Phase 5 (needed for load balancing key)
